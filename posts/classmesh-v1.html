ClassMesh v1: classifying a million log lines for the price of electricity

2026-07-31 · code on GitHub · Go, single binary, MIT

I spend my days building data pipelines, which means I spend a lot of my days looking at high volume streams of records that need to be sorted into buckets. Logs, events, rows. Some of them matter and most of them don't, and the whole game is deciding which is which without going broke or going slow.

ClassMesh is my answer to that. It's a classification cascade in Go: a stream of records goes in, each record exits at the cheapest stage that can decide it with confidence, and only the genuinely undecidable residue ever reaches anything expensive. Today I'm shipping v1. This post covers why it exists, how it works, the numbers, and what it honestly doesn't do yet.

Why I built it

There are two standard answers to "classify this firehose" and I dislike both.

The first is to throw a model at every record. Do the math on that. A million short log lines through a budget LLM API, at roughly 25 input and 5 output tokens each, costs about $6.75 per million lines, and it runs at API latency. That's fine for a demo. It's absurd for a pipeline that sees millions of lines an hour, and the dirty secret is that the overwhelming majority of those lines are health checks and noise that a string comparison could have handled.

The second answer is the hand rolled regex pipeline. Cheap, fast, and it rots. Nobody remembers why rule 14 exists, nobody dares delete it, and there's no concept of confidence or of what happens to records nothing matched. I've written several of these. I've regretted most of them.

The middle path seemed obvious once I said it out loud: run the cheap deterministic checks first, let everything they can confidently label exit immediately, and escalate only the leftovers to smarter, slower, pricier stages. Each stage has a confidence gate. A record that clears the gate exits. A record that doesn't gets passed down, and whatever survives every stage lands in a review sink instead of being silently dropped. The expensive stage still exists, it just sees a fraction of a percent of the traffic instead of all of it.

How it works

Everything in ClassMesh is an interface. Sources produce records, stages classify them, sinks receive them. Today the sources are text files, stdin, and JSONL; the stages are rules, schema checks, and a mock model stand-in; the sinks write JSONL. Tomorrow a source is anything that implements Source. The core engine is payload agnostic on purpose, because logs are the first use case, not the product.

A whole cascade is declared in one versioned YAML file:

version: 1
input:  { type: text }
stages:
  - { id: rules,  type: rules,  path: rules.yml, gate: 1.0 }
  - { id: schema, type: schema, path: schema.yml }
sink:   { type: jsonl, stream: stdout }
review: { type: jsonl, path: review.jsonl }

You can classmesh validate the config before anything opens an input file. Unknown keys, duplicate stage ids, and out of range gates are rejected up front. When the config declares category routes, classified records get dispatched by category, each route to its own sink, with the default sink as fallback. A top level workers: N classifies records on N goroutines while preserving output order, error reporting, and stats exactly. The default stays serial, because a default should be boring.

Every classified record comes out as one JSON object with its category, its confidence, which stage decided it, and the matched rule's reason. That last part matters to me. When rule 14 fires in this system, the output tells you rule 14 fired and why. The regex pipeline rot problem is mostly a legibility problem.

The numbers

Measured on a single core of an AMD Ryzen 7 3800X, with make bench in the repo, so you can check me:

PathPer recordThroughputAllocs
Rules stage, first-rule hit27 ns~37M rec/s0
Rules stage, worst case (20-rule walk, regex heavy)~7 µs~140k rec/s0
Full pipeline (engine + rules + sink)435 ns~2.3M rec/s0

The zero in the allocations column is the number I care most about. The hot path does not allocate. Not "rarely allocates", not "amortized". Zero. That's what keeps the throughput flat when the stream gets ugly, because there's no garbage collector waiting to present a bill at the worst moment.

Getting there was most of the work. The worst case path, where a record walks a rule list full of regexes and matches nothing, started around 7 microseconds and I got it down to about 1. Batch classification went from 7.6ms to 3.0ms for the same batch. None of it was clever. It was profiles, allocation traces, and restructuring code so the compiler could keep things on the stack. [EDIT: confirm these before/after numbers against your bench logs before publishing.]

Per record cost depends entirely on your ruleset. Order rules by expected volume so the hot path exits early. That single line of advice is worth more than any optimization in the engine.

Try it in 60 seconds

git clone https://github.com/ClassMesh/classmesh
cd classmesh
go run ./services/cli/cmd/classmesh run \
  --rules examples/rules.yml examples/logs.txt

That classifies the example logs with the example ruleset and prints one JSON object per record, with the undecided counted on stderr. The same binary handles JSONL events classified on their decoded fields with --input jsonl. No dependencies beyond Go.

The good

It's one static Go binary. There's nothing to install and nothing to deploy next to it. The interfaces made the whole thing a pleasure to test; the repo has more test files than I expected to write, and the benchmarks are committed right next to the code they measure, so the perf table above is a make target, not a screenshot.

The economics work exactly as designed. The rules stage handles the same million lines that would cost $6.75 through an API in well under a second per core, for the cost of the electricity. The cascade only forwards what rules can't decide to anything that costs money, which in most log streams is a small slice of the total.

The bad, honestly

The model stage isn't real yet. v1 ships rules, schema checks, the cascade config, category routing, and the parallel engine. The model tier is a deterministic mock that scores matched records with declared confidences below 1.0, which exists so per stage gates and review routing could be built and tested against something. The groundwork for a real model stage is in the repo, including a wordpiece tokenizer, but I'm not going to pretend a stand-in is a model. That's v2.

Sources are files and stdin. If you want Kafka, you're implementing Source yourself for now. And this is not a Datadog replacement or an observability platform. It's a classification engine you point at a stream. If you need dashboards, this is the thing that feeds them, not the thing that draws them.

What's next

v2 is the real model stage. I'm also working on something for the tier above that: running small models on the GPU from the same static Go binary, with no Python and no CUDA toolkit, through a pure Go CUDA driver binding I've been building called gocudrv. That's its own post.

Code: github.com/ClassMesh/classmesh. Issues and benchmark reproductions welcome. If the numbers don't hold on your hardware, I want to know.
